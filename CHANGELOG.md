# 更新日志 / Changelog

[中文 README](README.md) · [English README](README.en.md)

## [1.0.52] - 2026-05-22

### 修复 / 改进

- 重新登录/测试/查额度失败时，黑底日志区展示完整错误链（含阶段如「邮箱登录」「额度校验」、HTTP 403 等原文），不再只显示「重新登录失败」。
- 失败同时写入「运行日志」，便于后台排查。
- 顶部 toast 显示错误摘要（最多 160 字），不再吞掉 detail。

## [1.0.51] - 2026-05-22

### 改进

- 已分配账号：彻底跳过自动检测、自动查额度及后台导入检测队列；手动「开始测试」「查询额度」「重新登录」仍可用。

## [1.0.50] - 2026-05-22

### 改进

- 点击「重新登录」时，列表行状态/测试/额度列即时显示「登录中」并高亮该行，便于识别正在处理的账号。

## [1.0.49] - 2026-05-22

### 改进

- 定时自动测试与「立即测试」跳过**已分配**（已绑定卡密）的账号；仅在账号详情中手动「开始测试」/「重新登录」等操作会检测已分配账号。

## [1.0.48] - 2026-05-22

### 修复

- 修复加载「设置」时误清空 `proxyOptions`，导致账号详情代理下拉只剩「直连」的前端 bug。
- 打开账号详情加载测试选项时，自动将失效/空代理与当前代理池同步并重绘列表。

## [1.0.47] - 2026-05-22

### 修复

- 从设置中删除代理后，保存时自动清理账号上绑定的失效代理（有代理池则重新分配，无池则改直连）。
- 账号详情代理下拉框不再显示已删除的「xxx（当前）」项，仅展示当前代理池。

## [1.0.46] - 2026-05-22

### 修复 / 改进

- **重新登录**与重新导入对齐：清空 token、用库内素材刷新密码/令牌、应用详情里所选代理；403 等错误时自动切换代理池下一个代理重试。
- **重复导入**同一邮箱：更新凭证并重新走登录检测（不必先删账号）。
- 账号列表与详情中的**邮箱可点击复制**。

## [1.0.45] - 2026-05-22

### 修复

- 账号列表、卡密列表的「每页条数」写入浏览器本地存储，刷新页面后保持上次选择（如 100 条）。

## [1.0.44] - 2026-05-22

### 修复 / 改进

- 重新登录失败时回滚已写入的 token，修复「界面显示登录失败却能查额度/测试」的状态不一致。
- 重新登录成功后自动校验额度并执行模型测试（一次操作完成）。
- 失败提示区分「登录流程」与「额度校验」，不再笼统显示「登录失败」。

## [1.0.43] - 2026-05-22

### 修复

- **重新登录**（原「开始登录」）：清空旧 token → 未绑定则从代理池分配 → 完整 OTP/OAuth 登录 → 校验额度；403 时提示先配置代理。
- 保存代理池后自动给未绑定账号分配代理；展开账号时刷新代理下拉列表。

## [1.0.42] - 2026-05-22

### 修复

- 管理后台不再把英文技术错误（如 HTTP 403）统一显示为「操作失败，请稍后重试」；黑底日志区与 toast 展示真实原因与 `detail` 字段。

## [1.0.41] - 2026-05-22

### 新增

- 账号测试面板在「查询额度」前增加 **开始登录**：强制重新登录（邮箱 OTP / OAuth 刷新 token），用于掉登录、403 等异常恢复。

## [1.0.40] - 2026-05-22

### 新增

- 系统设置「自动测试」右侧增加 **立即测试** 按钮，手动触发全量 PP/GO 账号测试（与定时任务相同逻辑）。

## [1.0.39] - 2026-05-22

### 修复

- 移除 `docker-compose` 对 `./tools`、`./core` 的挂载（宿主机目录不完整时会覆盖镜像，导致 `session_converter_web.py` 找不到、502）。

## [1.0.38] - 2026-05-22

### 修复

- 移除 `docker-compose` 对 `./scripts:/app/scripts` 的挂载（覆盖 entrypoint 导致 Web 容器 `Restarting (2)`）；启动脚本改由镜像内提供，更新仍用 `/host-codex/scripts`。

## [1.0.37] - 2026-05-22

### 修复

- 更新脚本 `git reset` 改用 `FETCH_HEAD`（与服务器手动拉代码方式一致，避免卡在旧提交）。

## [1.0.36] - 2026-05-22

### 修复

- 移除 `docker-compose` 中 `./VERSION` 文件挂载（宿主机误为目录时会导致 `not a directory` 无法启动）；更新脚本自动修复误建的 `VERSION` 目录。

## [1.0.35] - 2026-05-22

### 修复

- **根因**：`HOST_BIND_PATH=/www/wwwroot/Codex` 在容器内用 `Path.is_dir()` 判断失败，仍误用 `/host-codex` 导致更新秒退。
- 新增 `scripts/trigger-update.py` 独立进程启动更新（日志含 `[engine] 1.0.35` 与 `[bind]`）；默认 `HOST_BIND_PATH`；复用当前 Web 镜像跑 compose。

## [1.0.34] - 2026-05-22

### 修复

- 一键更新根因：`docker run -v` 误用容器路径 `/host-codex`，宿主机上不存在导致任务秒退；现自动解析 mountinfo 或读取 `.env` 的 `HOST_BIND_PATH`（如 `/www/wwwroot/Codex`），并增加挂载预检。

## [1.0.33] - 2026-05-22

### 修复

- 一键更新改用 `docker:27-cli` 任务镜像（自带 `docker compose`），不再挂载宿主机 `/usr/local/bin/docker-compose`（宝塔常缺失导致秒退）。
- 更新秒退时把错误写入 `data/update-latest.log` 并在弹窗展示；修复 CRLF 脚本、compose 检测与健康检查。

## [1.0.32] - 2026-05-22

### 改进

- 后台「运行日志」改为终端代码风格（等宽深色 `<pre>`，时间/级别/分类/detail JSON 一行流展示）。

## [1.0.31] - 2026-05-22

### 文档 / 界面

- 说明一键更新架构：Web 容器只启动 `codex-update-job`，不在本容器内跑 git/compose。

## [1.0.30] - 2026-05-22

### 修复

- 一键更新重写：Web 容器只负责 `docker run` 拉起 **codex-update-job** 独立容器完成 git + compose；不再在即将被删除的 Web 容器里跑 git（解决卡住、502、日志重复）。

## [1.0.29] - 2026-05-22

### 修复

- 一键更新：`git fetch` 增加超时与浅克隆，避免卡在「拉取 GitHub」；日志不再重复两行。
- 重建后自动健康检查；前端在 502 期间继续轮询并提示「服务重启中」，勿中途刷新。

## [1.0.28] - 2026-05-22

## [1.0.27] - 2026-05-22
